# Database audit — 2026-06-12

> **Execution status: COMPLETE (merged to main as PR #37, 2026-06-12).**
> All five phases executed: Phase 0 (rate-semantics fix), Phase 1 (schema/
> validator hardening + vocabularies), Phase 2 (flagship refresh), Phase 3
> (full 127-card re-verification across 8 agent sweeps — every card now
> carries a 2026-06-12 verified/swept stamp), Phase 4 (tiered staleness
> budgets, last_swept_on, weekly URL-health CI). Per-card findings with
> sources and confidence levels: docs/verification-notes-2026-06.md.
> Remaining deferred items: docs/ROADMAP.md §"2026-06 audit follow-ups".

Full-dataset audit: 127 cards, 24 issuers, 5 networks, 5 loyalty programmes,
plus the schema, validator, and the site code that consumes the data.
Findings are severity-ranked; the remediation plan is at the bottom.

**Method.** Three passes:

1. Existing tooling — `scripts/validate.py`, `site/scripts/prebuild.mjs`,
   vitest suite (all pass; 3 pre-existing warnings).
2. A custom consistency sweep over every YAML file — temporal integrity,
   rate math, cross-file invariants, vocabulary drift — covering checks the
   validator does not run.
3. A web fact-check of 14 flagship cards (HDFC Infinia / DCB / Regalia Gold
   / Millennia / Tata Neu Infinity, Axis Magnus / Atlas / Reserve / Airtel,
   SBI Cashback, Amazon Pay ICICI, Amex Platinum Travel, IDFC FIRST Select,
   AU LIT) against issuer pages and card-tracking press as of 2026-06-12.

**Headline.** The dataset's *structure* is in excellent shape — zero schema
errors, zero overlapping or gapped effective-dated records, sources on every
record. The problems are one level up:

- **A semantics contract between data and code is broken** — the site
  mis-scores 127 of 158 accelerator rates across 86 cards (§A1).
- **The content is stale despite fresh verification stamps** — every one of
  the 14 fact-checked flagship cards has at least one material discrepancy,
  several with revisions *already in force* that the effective-dated arrays
  are missing (§B).

## A. Critical — data ⇄ code semantics breaks

### A1. `effective_rate` units: data says "points per ₹N", code says "percent"

The single biggest issue found.

- **Schema** (`card.schema.json`): `effective_rate` = "absolute rate
  per_inr" — i.e. units earned per the record's `per_inr` rupees.
- **Docs** (`DECISIONS.md`): "`effective_rate` stays as the documentary /
  receipt-visible total" — receipts show points, not percent.
- **Validator** (`validate.py`): the "`effective_rate > 5` with
  loyalty_program needs decomposition" error only makes sense under
  points-per-₹N semantics (the IndiGo 19/21/22 rates it was built for are
  points per ₹100).
- **Data**: authored as points-per-₹N throughout. Under that reading every
  open accelerator lands in a sane 0.2–15 % value band. Under a percent
  reading the data is absurd: Axis Reserve "45 % on dining", Magnus "35 %",
  BPCL Octane "25 % on fuel", Yes Marquee "24 % international".
- **Code**: all three consumers (`site/lib/calculator.ts:151`,
  `site/lib/detail-derivations.ts:20`, and downstream of
  `site/scripts/build.mjs`) treat `effective_rate` as an absolute percent
  and return it unconverted.

**Impact**: 158 open accelerators carry `effective_rate`; **127 of them, on
86 cards, are materially mis-scored** (everything where `unit_value ≠ 1` or
`per_inr ≠ 100`). The calculator and recommender inflate points-card
accelerators by 4–20×, and card detail pages render marketing copy like "It
earns up to 45% back on dining" (detail-derivations sentence 2) and wrong
"Accelerated earn rate of N%" pros. Cashback cards (`unit_value = 1`,
`per_inr = 100`) are correct only by coincidence — which is also why the
vitest suite (cashback-style fixtures only) passes.

**Root cause**: the comment at `calculator.ts:147-150` ("never multiply by
unit_value, that double-counts") shows the percent interpretation was
*introduced deliberately* at some point — the "5% entry with unit_value=0.25
became 1.25%" example it cites was actually the **correct** valuation of a
5-points-per-₹100 entry at ₹0.25/pt. The "fix" regressed the semantics, and
the rate-math primitive extraction (PR #36) then locked it in for all three
consumers. The schema's vague one-line description made this possible.

**Fix direction (recommended)**: keep the data as authored — schema, docs,
validator, and 127 files already agree — and fix the three code consumers to
convert via `pointsToPct(a.effective_rate, rewards.base.per_inr, unitValue)`.
Migrating the data to percent instead would churn ~90 files, destroy the
receipt-visible semantics the decomposition fields (`card_attributable_rate`,
`earn_components`) are defined against, and silently break the
`effective_rate > 5` validator rule. See plan Phase 0.

### A2. `multiplier` convention is inconsistent and self-contradicting

Schema/calculator semantics: `multiplier` × base rate. Marketing semantics
in India: "NX" relative to the issuer's 1-point baseline (which for cards
with `base.rate > 1` is *not* the base record). The data mixes both, and 14
open accelerators contradict their own `effective_rate`:

- `idfc-first/select` `online-spend`: `multiplier: 3`, `effective_rate: 6`,
  note says "6X" (base is 3/₹150 → ×3 would be 9, marketing 6X is 6/₹150).
  Same pattern on classic / millennia / wealth / first-private (10 records).
- `axis/atlas` + `axis/horizon` `travel-edge-portal`: `multiplier: 5`,
  `effective_rate: 5` — real rate is 5 mi/₹100 = 2.5× the 2/₹100 base.
- `axis/horizon` `international-spend`: `multiplier: 2`, `effective_rate: 2`
  (real: 1× base… the pair can't both be right).

Today the calculator ignores `multiplier` when `effective_rate` is present,
so these are dormant — but any consumer that prefers `multiplier` (or any
future refactor) flips 14 records to different values. Three accelerators
are multiplier-*only* and are live code paths now.

### A3. `headline_rate_pct` uses face value; calculator uses realized

`build.mjs computeHeadlineRatePct` computes from `unit_value_inr` (face) and
returns `null` if face is absent, while `calculator.ts unitValueFor` prefers
programme realized → `unit_value_inr_realized` → face. A card can be badged
2.0 % on listing pages but score 1.4 % in the recommender. Pick one basis
(realized, given ROADMAP's stance) for both.

  may deserve a dataset-level note rather than per-card.

## C. High — cross-file and per-file consistency

- **C1. Programme vs card unit values disagree.** Five cards pin
  `unit_value_inr_realized: 0.9` while their programmes say `0.95`
  (`rbl/irctc`, `hdfc/irctc-hdfc`, `sbi/irctc-premier`,
  `hdfc/tata-neu-infinity`, `hdfc/tata-neu-plus`). The calculator prefers
  the *programme* value, so the card-level numbers are dead-but-misleading.
  `irctc-loyalty.yaml` also disagrees with itself (`unit_value_inr.realized:
  0.95` vs `redemption_paths[0].rate_inr_per_unit.realized: 0.9`).
- **C2. `idfc-first/power` + `power-plus`**: `currency: cashback` with
  `unit_value_inr: 0.25` and `per_inr: 150` — that's a points card in a
  cashback coat (or the unit value is wrong). Either way the
  `is cashback ⇒ 1 unit = ₹1` invariant breaks for these two files.
- **C3. Flattened history (10 cards).** Records claim validity from launch
  but encode today's values: `axis/magnus` fees say ₹12,500 since 2019-07-01
  (launch fee was ₹10,000; raised Apr-2023) and benefits only start at the
  Sep-2023 nerf; HDFC Infinia / DCB / Regalia / Regalia Gold /
  diners-privilege benefits all start 2024 despite launches in 2009–2022.
  This is fine *if intentional* (history-before-X not modelled) but the
  records' `effective_from` values assert facts that are wrong. Decide a
  floor date policy (e.g. "history modelled from 2023-01-01; earlier
  `effective_from` must be accurate") and fix the offenders.

## D. Medium — vocabulary drift and completeness

- **D1. Free-form `accelerated[].merchants` is unvalidated.** 55 of 66
  distinct tokens are outside `channels/known.yaml`, with near-duplicates
  across the two fields: `eazydiner`/`eazy-diner`, `irctc`/`irctc-rail`/
  `irctc-direct`, `makemytrip`/`mmt`, `amazon`/`amazon.in`/`amazon-prime`.
  No code consumes the field today; it will bite the moment something does.
- **D2. `metadata.tags` has no vocabulary**: 94 distinct tags across 127
  cards with obvious synonym clusters — `mid`/`mid-tier`/`mid-premium`,
  `low-forex`/`zero-forex`/`forex-friendly`, `invite-like`/
  `invite-only-like`, `everyday`/`everyday-spend`, `grocery`(2) vs
  `groceries`(0). Filter facets built on tags will fragment.
- **D3. 18 open accelerators lack `canonical_categories`** and dodge the
  regex lint because their `category` strings don't match any rule —
  including inconsistent twins: `idfc-first/wealth` tags
  `online-spend-above-20k` as `[online]` while select/millennia/classic
  leave the identical accelerator untagged (invisible to the calculator).
  Others (`magnus` monthly-threshold, `onecard` top-2-categories,
  `au/lit` toggle) are genuinely unbucketable and should be listed in
  `untaggable` so the gap is explicit.
- **D4. 9 channel tokens are declared but unused anywhere** (`bbps`,
  `eazy-diner`, `fuel-network-any`, `irctc-rail`, `pos-any`, `reliance-bp`,
  `reliance-retail`, `shell`, `swiggy-one`) — two of them (`eazy-diner`,
  `irctc-rail`) exist because the card files used a different spelling in
  the unvalidated field (D1).
- **D5. `image_path` missing on 126/127** cards (only placeholder rendering
  ships). Known gap per `ASSETS.md`; listed for completeness.
- **D6. Source hygiene gaps**: aggregator URLs (`paisabazaar`, `cardinsider`)
  appear as *nested* record sources on active axis/hdfc/idfc/kotak/sbi cards
  — the validator only inspects the first source in the file. The
  issuer-domain allowlist covers 8 of 24 issuers. One IDFC record cites
  `business-standard.com` (fine as secondary, but it's a primary on a record).

Verified clean (for the record): no future dates, no dated-array gaps or
overlaps, no empty benefit records, no missing milestone/welcome values, no
cap/cycle omissions, every active card has forex + waiver data, all six
apply-url-less cards are invite-only (expected).

## E. Why this slipped through — schema & validator gaps

1. `effective_rate` has no units anchor in the schema — one ambiguous
   description line is the entire contract (enabled A1).
2. No lint relates `multiplier`, `base.rate`, and `effective_rate` (A2 would
   have been caught: `base.rate × multiplier ≠ effective_rate` on 14 records).
3. No plausibility band on computed accelerator *value* (a "45 % dining"
   record should be an error without decomposition, like the existing
   `> 5 with loyalty_program` rule but value-based and universal).
4. No cross-check of card `unit_value_inr_realized` vs the referenced
   programme's realized value (C1).
5. `currency: cashback` does not force `unit_value_inr: 1` (C2).
6. Staleness checks are repo-wide (180 d) and trust `last_verified_on`
   blindly; nothing distinguishes "verified" from "bulk-stamped" (B).
7. Discontinued/on-hold lifecycle: no check that a discontinued card's
   records are all closed (latent; zero offenders only because there's one
   discontinued card).
8. `accelerated[].merchants` and `metadata.tags` are uncontrolled
   vocabularies (D1, D2).
9. Aggregator-source warning only fires on the file's first source; issuer
   allowlist covers a third of issuers (D6).
10. Tests validate code-vs-code, not code-vs-data-semantics: all fixtures
    are cashback-shaped, where the A1 ambiguity is invisible.

# Remediation plan

Ordered phases; each is an independent PR (or small PR series) and leaves
the repo green. Phases 0–1 are code/schema; 2–3 are data; 4 is process.

## Phase 0 — Fix the rate semantics regression (code, ~1 day) ⚠ ship first

1. **Decide and document the contract**: `effective_rate` = units per the
   parent record's `base.per_inr` rupees (receipt-visible). Encode it in the
   schema description, `SCHEMA.md`, and a `DECISIONS.md` entry superseding
   the percent interpretation.
2. `calculator.ts acceleratorRatePct`: replace `return a.effective_rate`
   with `pointsToPct(a.effective_rate, rewards.base.per_inr, unitValue)`
   (null-guard unitValue; cashback unit value is 1).
3. `detail-derivations.ts`: same conversion in `effectivePctOf`; fix the
   user-visible copy — "up to N%" must use the converted value, and
   `formatAccelerated` should render points-card rates as "N pts/₹100", not
   "N%". The `>= 5` threshold in `derivePros` becomes a value-% threshold.
4. Add vitest fixtures that *break under percent semantics*: a points card
   with `effective_rate: 10, per_inr: 150, unit_value: 0.22` must score
   1.47 %, not 10 %. Add an Axis-Reserve-shaped regression case (45/₹200 @
   0.35 → 7.875 %).
5. Re-run `build.mjs`, spot-check `dist/cards.json` and 3–4 detail pages.
6. Same PR or follow-up: A3 — make `computeHeadlineRatePct` use the
   calculator's `unitValueFor` preference order (realized first).

## Phase 1 — Schema + validator hardening (~2 days)

1. **Multiplier cleanup (A2)**: relax schema so `accelerated` requires
   `multiplier` *or* `effective_rate` (not always `multiplier`); then drop
   the contradicting/dummy multiplier values from the 14 records (data fix
   in the same PR, validator lint `base.rate × multiplier ≈ effective_rate
   when both present` promoted to error).
2. New validator errors (all have zero or few known offenders, so promote
   immediately after the offending files are fixed):
   - computed accelerator value-% > 12 without `earn_components` /
     `card_attributable_rate` decomposition;
   - card realized unit value ≠ referenced programme realized (C1);
   - `currency: cashback` ⇒ `unit_value_inr` ∈ {null, 1} (C2);
   - discontinued ⇒ all records closed; `effective_until ≤ discontinued_on`;
   - dated-array *gap* detection (currently zero offenders — cheap insurance);
   - aggregator check across **all** sources of active cards (downgrade to
     per-record warning to avoid blocking history records);
   - extend `ISSUER_ALLOWED_DOMAINS` to all 24 issuers.
3. **Vocabularies**: add `data/tags.yaml` (controlled list + alias map) and
   fold `accelerated[].merchants` into the channel-token vocabulary (one
   namespace, one spelling per merchant); validator enforces both. Merge the
   synonym tags (D2) and reconcile `eazydiner`/`irctc` spellings (D1, D4) in
   the same PR.
4. Tag the 15 taggable accelerators from D3; add the 3 genuinely
   unbucketable patterns to `category_rules.yaml`'s `untaggable` regex.

## Phase 2 — Flagship refresh wave (data, ~3–4 days)

Apply the §B table. For each card: close the stale open record with the
real-world `effective_until`, append a new open record with the revision's
effective date and an issuer-owned source. Priorities:

1. **Already-wrong-today, high-traffic** (do first): `sbi/cashback`,
   `axis/airtel`, `icici/amazon-pay`, `au/lit` (forex + lounge), `hdfc/
   millennia` (lounge), `hdfc/infinia` + `hdfc/diners-black` (SmartBuy 5X,
   redemption caps), `amex/platinum-travel` (milestones).
2. **Status + history**: `axis/atlas` → `on-hold` (closed to new
   applicants; note the no-official-statement caveat in the record),
   milestone/transfer-cap corrections; `axis/magnus` base-rate and fee
   history (C3); `axis/reserve` waiver + accelerator.
3. **Announced, dated future change**: `hdfc/regalia-gold` Jul-2026 lounge
   record (the effective-dated model exists precisely for this).
4. Fix C3 flattened-history offenders alongside (decide the history-floor
   policy first; document it in `CONTRIBUTING.md`).

Definition of done per card: every open record's source is issuer-owned or
archived-issuer, `last_verified_on` set to the day a human/agent actually
diffed the fields against the source — never bulk-stamped.

## Phase 3 — Full-catalogue re-verification sweep (data, ~1–2 weeks elapsed)

The 14/14 flagship hit rate means the remaining 113 cards need the same
treatment, batched by issuer in descending traffic order (HDFC → Axis →
SBI → ICICI → IDFC FIRST → Amex → Kotak → AU → RBL → IndusInd → rest).
Combine with the two ROADMAP debts so each file is touched once:

- replace per-issuer-haircut realized values with sourced numbers
  (second-tier issuers, ~50 cards);
- verify `card_attributable_rate` on co-brand cards against rate-card PDFs.

Track as a checklist PR series (`docs/superpowers/plans/` pattern), ~10–15
cards per PR so review stays tractable.

## Phase 4 — Keep it from rotting (process)

1. **Staleness budget by tier**: tighten `STALENESS_WARN_DAYS` to 90 for
   `active` super-premium/premium cards (highest churn + traffic), keep 180
   for the rest; fail CI at 270.
2. **Devaluation watch**: a lightweight weekly CI job (or recurring agent
   task) that greps a small set of feeds (issuer T&C pages, CardExpert /
   CardInsider / LiveFromALounge indexes) for the 30 highest-traffic card
   names and opens an issue per hit. Devaluations land with 30–60 day
   notice — the effective-dated model can capture them *before* they bite,
   as Phase 2.3 does for Regalia Gold.
3. **Verification honesty rule** in `CONTRIBUTING.md`: `last_verified_on`
   may only move when the record's fields were actually compared against the
   cited source; bulk refreshes touch a new `metadata.last_swept_on` field
   instead (schema addition).
4. Re-run the URL reachability check (`--check-urls`) on a weekly schedule
   instead of never (it's opt-in today and not wired into CI).
